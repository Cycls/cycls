# RFC003 — Share & Collaboration

## Summary

Replace the HMAC signed-URL share mechanism with opaque tokens + per-tenant DB lookups. Add file-as-first-class-resource sharing, org-private audiences, and forks (deep copy). Drop `signing.py` and `App.signing_key` entirely. App becomes stateless w.r.t. its volume.

## Motivation

Current share machinery (RFC002 era):
- HMAC signed URLs (S3-style) for chats only
- Files only shareable as attachments inside chat shares
- "Delete share" lies — removes the owner-UI row but the signed URL keeps working until `exp`
- Multi-param URLs: `/shared?path=&user=&exp=&sig=` (~130 chars, four params)
- No org-private; no forks; no genuine revocation

Sharing is a core value prop of cycls beyond "ChatGPT on a drive." It deserves a coherent design pass, not another ad-hoc addition.

## Goals (v1)

- Share chats and files as first-class resources
- Audiences: `public` (link) and `org:<oid>` (authenticated org member)
- Forks via deep copy
- Real revocation (delete row → URL stops working)
- Clean URL: `/share/<user>/<token>`
- Lowest LOC that doesn't paint into a corner

## Non-goals (v1)

- Real-time collaboration (no CRDT, no presence, no websockets)
- "Shared with me" inbox view
- Folders as shareable (foundation must allow extension)
- Agent sharing
- Comment / edit / finer-grained roles
- Stable file IDs (path-as-ID for v1; rename breaks shares — documented limitation)
- Audit log
- View tracking / analytics

## Design

### Resource model

Two first-class shareable kinds, addressed by path within a workspace:

```
chat/<chat_id>
file/<path>
```

(Folders, projects, agents — future, same machinery.)

### Principals and roles

Implicit two-role: `view` (anyone in the audience) and `own` (the workspace's tenant). Owner is `subject_for(user)` of the workspace.

Audiences, encoded in the share token row:

```
public          — anyone with the URL
org:<oid>       — authenticated user where user.org_id == oid
```

(Future: `user:<uid>` for direct shares; same field, more cases.)

### Share tokens

A share is a row in the **owner's workspace** at `share/<token>`:

```python
{
    "path":       "chat/abc" | "file/notes/journal.md",
    "exp":        1735689600,
    "audience":   "public" | "org:acme",
    "created_at": "2026-05-03T...",
}
```

`token = secrets.token_urlsafe(16)` → ~22-char URL-safe random.

### URL shape

```
/share/<user_subject>/<token>
```

Example: `/share/u_456/aBcDeFgHiJkLmN-Op_QrSt`

The user subject in the URL is `subject_for(user)` (today's `u_id` or `org_id/u_id`). Lookup is per-tenant: `DB(workspace_at(user, ...)).get(f"share/{token}")`. **No deployment-level DB in v1.** The user-in-URL isn't a secret — same info as today's `&user=` query param.

### Resolution flow

```
GET /share/<user>/<token>
```

1. Look up `share/<token>` in `workspace_at(user, ...)`. Missing or `exp < now` → 404.
2. Check audience:
   - `public` → proceed
   - `org:<oid>` → require auth (existing `validator`); if absent → 401; if `request.user.org_id != oid` → 403; else proceed
3. Dispatch on `path` prefix:
   - `chat/<id>` → load chat metadata + messages from owner's workspace; for each attachment, mint a sub-token (same `exp`, same `audience`) so the SPA can load files natively
   - `file/<path>` → resolve path-traversal-safely inside `workspace.root`; return `FileResponse`
4. Return.

### Forks

```
POST /share/<user>/<token>/fork
```

1. Resolve the share (audience check). Caller must have view access.
2. Read source: chat metadata, messages, attached files.
3. Generate new resource IDs in the **forking user's** workspace.
4. Deep-copy the data; rewrite attachment paths.
5. Return the fork's own resource URL.

The fork is independent. Source can be deleted; fork survives. No cross-tenant runtime relationship.

Optional in v1: source row records `forks: [<fork_owner_subject>, ...]` for "X people forked this" — provenance only, no access coupling. Defer to v2 if not needed for UI yet.

### Owner-side operations

```
POST   /share          — mint a token. body: {path, audience, ttl?}
GET    /share          — list this owner's tokens (scan share/* in their workspace)
DELETE /share/<token>  — revoke. The row is deleted; URL stops working immediately.
```

Owner-side display metadata (title override, etc.) — *not in v1*. The chat already has its title; the file already has its name. Don't store redundancy.

### Storage layout

Per workspace, no new deployment-level scope:

```
<workspace>/
    chat/...                  (existing)
    file/...                  (existing — under workspace.root)
    share/<token>             (new — share tokens)
```

### Removed surface

- `cycls/app/signing.py` — entire file deleted
- `App.signing_key` cached_property
- `App.signed_url` and `App.verify_signed` methods
- `<volume>/.cycls/signing.key` file (App stops writing to disk)
- Existing `/shared` and `/shared/file/<path>` routes
- Existing `share/<chat_id>` DB rows (different shape from the new `share/<token>`)

App becomes stateless w.r.t. its volume. The only writes to `<volume>` are now user-driven via DB.

### Net LOC estimate

- Drop: signing.py (31) + App.signing_key (5) + signed_url (10) + verify_signed (2) + old share routes (~80) ≈ **−128 LOC**
- Add: new share routes + fork route + token mint/revoke ≈ **+90 LOC**
- **Net: ~−40 LOC** with strictly more product surface (file shares, org-private, real revoke, forks).

## Migration

`rfc-002-kv` is pre-launch; no production deployments. **Replace in place.** Existing share URLs become invalid at deploy; acceptable.

If RFC003 lands on a deployed system in the future, ship a transitional period where both URL shapes resolve.

## Extension points (designed-in, not built)

- **"Shared with me" inbox.** Add a deployment-level grants index keyed by principal. Dual-write on share creation. New URL: `/inbox` returns scan of the index. Today's per-tenant model unchanged.
- **Folders.** New resource kind `folder/<path>`. Same share machinery; resolver lists files inside on resolution.
- **Stable file IDs.** Add per-workspace `files/<id> → <path>` registry. Share rows store `id` instead of literal path. Migration: rewrite share rows. URL format unchanged.
- **Finer roles.** Add `role` field to share row: `"view" | "comment" | "edit"`. Resolver gates routes by role.
- **Direct user shares.** New audience `user:<uid>`. Resolver requires auth and checks `user.id == uid`.
- **Audit log.** Deployment-level append-only `audit/` namespace. Write on every share resolution.
- **Real-time collab.** Out of scope; would require a separate websocket/CRDT layer atop this storage model.

## Open questions

1. **Attachment URL strategy.** Pre-mint sub-tokens (one DB write per attachment) vs serve attachments under the parent share path (`/share/<user>/<chat-token>/file/<path>` with server validating it's an attachment of that chat). Both work. Settle during implementation; sub-token approach is the safer default — symmetry with the file-share path.
2. **Org-membership source.** RFC003 assumes `User.org_id` from the JWT claim is authoritative. If cycls later supports multi-org users or sub-groups, this becomes a separate auth question, not a share question.
3. **Fork attribution UI.** Do we surface `forked_from` to the fork owner? To the source owner? Defer until product UI design.

## FE follow-ups

Backend share API is complete; FE wired. Gaps closed: file-share audience picker (public / org-private), error feedback on share-mint and fork failures, a fork in-flight indicator, and `?fork=` now survives the OAuth sign-in round-trip (added to `PERSIST_KEYS`).

**Deferred (polish, low value):** show the actual URL on file-share success (conflicts with the toast aesthetic we settled on); "Forked from \<author\>" attribution in the shared/forked view (`forked_from` exists in metadata, unrendered).

**Decided against:**
- **Per-share expiry / TTL picker** — shares don't expire by default; the delete button (everything's listed in "Manage shares") is the control. The API keeps an optional `ttl` for short-lived links, but the FE deliberately doesn't surface a picker — Drive-style control, no demand, surprising failure mode ("my link broke?"). Don't re-add it.
- **Edit share** (change audience/ttl after creation) — API doesn't support it.
- **Reactivate revoked share** — `DELETE` removes the row; no resurrection.
- **Share usage analytics** ("viewed N times") — needs a deployment-level audit log (see Extension points).

## Status

Proposal. The architectural decisions are made; awaiting approval to implement.
